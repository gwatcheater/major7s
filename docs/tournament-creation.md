# Manual New Tournament Creation

Technical reference for the admin-driven workflow that adds a new major
tournament to Major7s. Covers the UI form, the trigger action, the
background side-effects, and the resulting database state.

---

## 1. Process Overview

A site admin opens the **Admin → Tournament** tab and expands the
**Create New Tournament** card. They pick the major, confirm the
auto-derived dates and deadline, optionally add logo/external URLs, then
click **Create Tournament**. The client performs a single authenticated
`INSERT` into `public.tournaments` using the browser Supabase client.
Postgres generates the primary key, applies column defaults
(including the standard 7-bucket layout), and `set_updated_at` keeps
`updated_at` in sync on later edits. React Query caches for the admin
list and the public tournament feeds are then invalidated so the new
record appears immediately across the app.

No edge function, server function, or external API call is involved in
the create step itself — golfer field setup, ESPN linking and picks
are handled later from the **Field Management** screen.

## 2. User Input & Manual Field Population

Form location: `src/routes/_authenticated/admin.index.tsx` →
`CreateTournamentForm` (lines 564–789).

| Field | UI control | Type | Required | Purpose / Notes |
|---|---|---|---|---|
| **Name** | `<select>` dropdown | enum-like text (one of four majors) | Yes | Sets `tournaments.name`. Options: *Masters Tournament*, *PGA Championship*, *U.S. Open*, *The Open Championship*. Selecting *Masters Tournament* auto-fills `location` with `Augusta National Golf Club`; switching away clears it. |
| **Location** | `<input type="text">` (max 120 chars) | text | Yes | Course / venue. Free text; pre-filled for the Masters. |
| **Logo URL** | `<input type="url">` | text (nullable) | No | Public image URL used on tournament cards. Stored as `NULL` when blank. |
| **External Link URL** | `<input type="url">` | text (nullable) | No | Official tournament site or live leaderboard. Surfaces as the "Live Leaderboard" link on the tournament view. |
| **Start Date** | `<input type="date">` | ISO date (`YYYY-MM-DD`) | Yes | Tournament Thursday. On change, the form auto-populates *End Date* (start + 4 days) and *Submission Deadline* (start − 1 day @ 22:00 local) using local-time arithmetic to avoid UTC date-shift bugs. |
| **End Date** | `<input type="date">` | ISO date | Yes | Sunday of tournament week. Auto-derived from Start Date but editable. |
| **Submission Deadline** | `<input type="datetime-local">` | local datetime | Yes | Picks lock-time. Converted to an ISO UTC string (`new Date(value).toISOString()`) before insert. Default = Wednesday 22:00 local. |
| **Status** | `<select>` dropdown | enum `public.tournament_status` | Yes (defaults `upcoming`) | One of `upcoming`, `live`, `completed` (whatever keys are present in `TSTATUS_LABEL`). Drives visibility on the home / archive feeds. |

Client-side validation: the submit handler rejects the form when any of
*name*, *location*, *start_date*, *end_date* or *submission_deadline* are
empty, surfacing a `toast.error("Fill in all required fields")`.

## 3. Admin Trigger Actions

| Element | Behaviour |
|---|---|
| **Show ▼ / Hide ▲** in the `CardHeader` | Toggles the form's `open` local state — purely cosmetic, no DB effect. |
| **Create Tournament** button (`<Button type="submit">`) | Fires the form's `onSubmit` → `submit(e)` handler. Disabled while `saving === true` and re-labelled `Creating…`. |

The submit handler is the only mutation entry point. There is no
multi-step wizard, no draft persistence, and no separate "publish" action.

---

## 4. Automated Background Processes & System IDs

When **Create Tournament** is clicked, the following happens — all
inside the browser tab against the public Supabase Data API:

1. **Field validation** — required fields checked client-side.
2. **Payload assembly** — the form values are normalised:
   - `logo_url` / `external_url` → `null` when blank.
   - `submission_deadline` → `new Date(value).toISOString()` (UTC).
3. **Insert call** — single round-trip:
   ```ts
   supabase.from("tournaments")
     .insert({ name, location, logo_url, external_url,
               start_date, end_date, submission_deadline, status })
     .select("id")
     .single();
   ```
4. **Postgres-side automatic population** (no app code runs here; these
   are column defaults defined in the schema):
   - `id` — `uuid` generated by `gen_random_uuid()` (the **Tournament ID**, the canonical foreign key everywhere else).
   - `bucket_sizes` — `jsonb` defaulted to
     `{"1":10,"2":10,"3":10,"4":10,"5":0,"6":0,"7":0}` (the seven OWGR-ranked picks buckets).
   - `status` — defaults to `'upcoming'` if not provided.
   - `created_at`, `updated_at` — `now()`.
5. **Returned ID** — the new `id` is read via `.select("id").single()` and passed to `onCreated(data.id)`.
6. **Toast + form reset** — success toast, state reset, form ready for another create.
7. **Cache invalidation** (in the parent `TournamentAdminTab` `onCreated`):
   - `["admin-tournaments-list"]` — admin selector.
   - `["tournaments-active"]` — public live/upcoming feed.
   - `["tournaments-completed"]` — archive feed.
8. **Auto-selection** — the parent sets `selectedId` to the new tournament so its **Field Management**, **Edit Details**, **ESPN Leaderboard**, **Bulk Picks Upload** sub-cards render immediately.

> Note: there are **no Round IDs or Stage IDs** in this system. A tournament is a single row; rounds 1–4 are tracked implicitly via `tournament_scores.round` integer columns once leaderboard data is imported. Golfer-pool rows (`golfers`) and pick rows (`picks`) get their own `gen_random_uuid()` PKs only when those entities are later created from the Field Management screen.

There are **no DB triggers** firing on `tournaments` insert (see
`<db-triggers>`: none). No emails, webhooks, or pg_cron jobs are
scheduled at create time.

## 5. Data Interaction & Schema Mapping

### Target table: `public.tournaments`

| Column | Type | Source | Notes |
|---|---|---|---|
| `id` | `uuid` PK | `gen_random_uuid()` | **Tournament ID** — referenced by every downstream table. |
| `name` | `text` NOT NULL | Form: Name | One of four major strings. |
| `location` | `text` NOT NULL | Form: Location | |
| `logo_url` | `text` NULL | Form: Logo URL | |
| `external_url` | `text` NULL | Form: External Link URL | |
| `start_date` | `date` NOT NULL | Form: Start Date | |
| `end_date` | `date` NOT NULL | Form: End Date | |
| `submission_deadline` | `timestamptz` NOT NULL | Form: Submission Deadline (UTC) | Enforced by `enforce_pick_lock()` trigger on `picks`. |
| `status` | `tournament_status` NOT NULL | Form: Status | Defaults `'upcoming'`. |
| `bucket_sizes` | `jsonb` NOT NULL | DB default | `{1:10,2:10,3:10,4:10,5:0,6:0,7:0}` — editable later in Field Management. |
| `espn_event_id` | `text` NULL | — | Populated later by the ESPN Leaderboard section. |
| `recap_blog` | `text` NULL | — | Set later from the blog editor. |
| `created_at` / `updated_at` | `timestamptz` NOT NULL | `now()` | `updated_at` refreshed by `set_updated_at` trigger on subsequent updates. |

### Downstream foreign-key relationships (populated later, not at create)

```text
tournaments.id  ──┬──< golfers.tournament_id            (field pool, bucket 1–7)
                  ├──< picks.tournament_id              (user picks, gated by submission_deadline)
                  ├──< tournament_leaderboard.tournament_id
                  ├──< tournament_results.tournament_id
                  ├──< tournament_scores.tournament_id  (round-level scores)
                  ├──< tournament_score_picks.tournament_id
                  └──< blog_posts.tournament_id         (recaps / previews)
```

A freshly-created tournament therefore exists as a **single isolated row**.
All linked entities (`golfers`, `picks`, leaderboard/score tables,
recap `blog_posts`) are inserted by separate admin actions:

1. **Field Management** (`/admin/tournament/$id/field`) — admins add
   `golfers` rows with `tournament_id = <new id>` and assign
   `bucket_number ∈ 1..7`. `bucket_sizes` governs how many per bucket.
2. **ESPN Leaderboard section** — patches `espn_event_id` on the
   tournament row and seeds the `tournament_leaderboard` /
   `tournament_scores` tables on demand.
3. **User picks** — players write into `public.picks` with the
   `tournament_id`; the `enforce_pick_lock()` SECURITY DEFINER trigger
   reads `tournaments.submission_deadline` to allow / reject writes
   (admins bypass via `has_role(auth.uid(), 'admin')`).
4. **Picks-confirmation email** — fired from the lineup save handler
   (`useServerFn(sendPicksConfirmation)`), not from tournament creation.

### Access control

- Insert is performed with the user's session via
  `@/integrations/supabase/client`. RLS on `tournaments` restricts
  writes to admins (`public.has_role(auth.uid(), 'admin')`).
- No service-role key is used for create — the admin's JWT is sufficient.
