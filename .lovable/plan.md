## Approval email — plan

Uses the existing Lovable Emails transactional pipeline (same as `recovery-link`, `migration-welcome`, `picks-confirmation`). No Resend, no new API keys.

**Confirmed send pattern** (from `sendRecoveryLinks` in `src/lib/admin-users.functions.ts` lines 548–700):
- `supabaseAdmin.rpc("enqueue_email", { queue_name: "transactional_emails", payload })` — direct RPC, NOT a fetch to `/lovable/email/transactional/send`.
- Full pipeline is manual: render React Email → `toPlainText(html)` → resolve subject → mint `messageId` + `idempotency_key: <label>:<id>:<Date.now()>` → get-or-create unsubscribe token → insert `email_send_log` `pending` row → enqueue → on enqueue error insert `failed` row.
- `sendApprovalEmail` will mirror this exactly.

- React Email template, brand tokens matching `recovery-link.tsx` (Forest `#103D2E`, Gold `#C9A227` CTA, white body, Arial stack, rounded container).
- Props: `firstName?: string`, `loginUrl: string` (caller passes `${getPublicSiteOrigin()}/login`).

**MODIFY `src/lib/email-templates/registry.ts`**

**MODIFY `src/lib/admin-users.functions.ts`**
Add `sendApprovalEmail` server fn — single-user version of `sendRecoveryLinks`:
- `createServerFn({ method: "POST" }).middleware([requireSupabaseAuth])`, `assertAdmin(context)`.
- Input: `z.object({ userId: z.string().uuid() })`.
- Load profile (`id, email, first_name`).
- Suppression check via `supabaseAdmin.from('suppressed_emails')`. If suppressed: write `admin_audit` row `action: 'user.approved_email_skipped'` with `detail: { user_id, reason: 'suppressed' }`, return `{ ok: false, error: 'suppressed' }`.
- Render `user-approved` template with `firstName` and `loginUrl = ${getPublicSiteOrigin()}/login`; compute subject.
- `idempotency_key: user-approved:${userId}:${Date.now()}` — timestamped so re-approval after demotion re-sends. (Enqueue's dedup is not time-boxed at the app layer; keeping timestamp keeps behavior consistent with `sendRecoveryLinks`.)
- Mint `messageId`, get-or-create unsubscribe token (same block as `sendRecoveryLinks`), insert `email_send_log` `pending`, `enqueue_email` into `transactional_emails` with `label: 'user-approved'`.
- On enqueue error: insert `email_send_log` `failed` row, write `admin_audit` `action: 'user.approved_email_failed'` with `detail: { user_id, error }`, return `{ ok: false, error }`.
- On success: write `admin_audit` `action: 'user.approved_email_sent'` with `detail: { user_id }`, return `{ ok: true }`.
- Never throws — status update in the UI must not roll back on email failure.

**MODIFY `src/components/admin/users-directory-tab.tsx`** (no UI change)
- At top of component: `const sendApproval = useServerFn(sendApprovalEmail);` (import added).
- In `handleStatusChange` (line 869): capture `const prevStatus = status;` before the update. Status update stays first and independent. After success toast + invalidations, add:

- No changes elsewhere: registration, `bulkCreate`, `sendWelcomeEmails`, admin approval UI layout all untouched.

### Guards / non-goals

- No DB trigger — `bulkCreate` inserts users straight as `approved`, and a trigger would spam those. Scoping to the UI handler is intentional.
- Not wired into `handle_new_user`, `notify_admin_on_new_profile`, `bulkCreate`, `sendWelcomeEmails`.
- No auth-settings or registration changes.
